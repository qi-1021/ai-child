## 2️⃣ 知识查询和注入的具体代码

### 关键代码片段 1: 知识查询逻辑

**文件**: [server/ai/child.py](server/ai/child.py#L171-L189)

```python
async def chat(
    session: AsyncSession,
    user_text: str,
    content_type: str = "text",
    media_path: Optional[str] = None,
) -> Tuple[str, Optional[str]]:
    # ... 代码省略 ...
    
    # 第二步: 构建上下文
    name = await get_ai_name(session)
    profile = await get_or_create_profile(session)
    is_sleeping = profile.is_sleeping
    history = await get_recent_messages(session, limit=settings.memory_context_turns)
    
    # 📍 获取所有知识
    all_knowledge = await get_all_knowledge(session)

    # 📍 搜索相关知识（通过用户输入的前64字符）
    relevant = await search_knowledge(session, user_text[:64])
    seen_ids = {k.id for k in relevant}
    
    # 📍 关键：将所有未搜索到的知识也加入
    for k in all_knowledge:
        if k.id not in seen_ids:
            relevant.append(k)

    # 构建消息列表（包含所有知识）
    messages = _build_context(history, relevant, name, is_sleeping)
```

**问题分析**:
- ❌ `search_knowledge(user_text[:64])` 只用前64字符做搜索（可能导致准确率不高）
- ❌ 最后的循环逻辑使得 `relevant` 实际上包含**所有**知识，搜索结果被忽略
- ❌ 没有任何筛选、排序或优先级机制

### 关键代码片段 2: 知识注入到系统提示词

**文件**: [server/ai/child.py](server/ai/child.py#L105-L130)

```python
def _build_context(
    history: List[Conversation],
    knowledge_items: List,
    name: str | None,
    is_sleeping: bool = False,
) -> List[Dict[str, Any]]:
    """Build the OpenAI messages list from DB history and knowledge."""
    messages: List[Dict[str, Any]] = [
        {"role": "system", "content": _build_system_prompt(name, is_sleeping)}
    ]

    # 📍 知识注入点
    if knowledge_items:
        kb_text = "\n".join(
            f"- [{item.topic}] {item.content}" for item in knowledge_items
        )
        messages.append(
            {
                "role": "system",
                "content": f"Things you have learned so far:\n{kb_text}",
            }
        )

    # 对话历史
    for msg in history:
        messages.append({"role": msg.role, "content": msg.content})

    return messages
```

**格式分析**:
```
系统提示词 (Message 1):
- 身份信息: "You are {name}, an AI child."
- 行为指南: 长段落 (~500-800 tokens)

知识块 (Message 2):
- 格式: "Things you have learned so far:"
- 内容: "- [Topic1] Content1\n- [Topic2] Content2\n..."
- 每条知识: `[{topic}] {content}` (固定格式)

对话历史 (Messages 3+):
- 用户消息
- 助手回复
- 工具调用结果
```

## 4️⃣ 单次注入的 Token 成本分析

### 成本分解

```
┌─────────────────────────────────────────────────────────┐
│         单次 chat() 调用的完整 Token 成本               │
└─────────────────────────────────────────────────────────┘

1. 系统提示词 (System Prompt #1)
   ├─ 身份信息 (Identity)
   │  └─ "You are {name}, an AI child." (40-50 tokens)
   ├─ 行为指南 (Behavioral Guidelines)
   │  └─ "You are warm, genuine, intensely curious..." (520-620 tokens)
   ├─ 防幻觉指南 (Anti-hallucination Guidelines)
   │  └─ "When uncertain or making important claims..." (180-250 tokens)
   └─ 睡眠模式说明 (Sleep Mode Notes) [如果在睡眠中]
      └─ "You are currently in rest/sleep mode..." (50-80 tokens)
   
   💰 小计: 500-800 tokens

2. 知识块 (Knowledge Block #2 - System Role)
   └─ "Things you have learned so far:"
      ├─ 开头文本 (5-10 tokens)
      └─ 知识列表 (N 条 × 10-50 tokens/条)
          └─ 格式: "- [Topic] Content\n"
   
   💰 单个知识: ~15-60 tokens
   💰 总计 (N知识): N × 20-50 tokens
   
   📌 典型场景:
      • 5条知识: 100-250 tokens
      • 20条知识: 400-1000 tokens
      • 50条知识: 1000-2500 tokens

3. 对话历史 (Conversation History)
   ├─ 用户消息 #1: 20-100 tokens
   ├─ 助手回复 #1: 50-300 tokens
   ├─ 用户消息 #2: 20-100 tokens
   ├─ 助手回复 #2: 50-300 tokens
   └─ ... (20 轮 = 40 条消息)
   
   💰 单条消息: ~20-100 tokens (平均 40 tokens)
   💰 总计 (20轮): 40消息 × 40 tokens = 1600 tokens
   💰 范围: 1000-2500 tokens

4. 用户当前消息 (Current User Input)
   └─ user_text
   
   💰 10-200 tokens (取决于长度)

┌─────────────────────────────────────────────────────────┐
│                    TOTAL FOR ONE CALL                   │
├─────────────────────────────────────────────────────────┤
│ 系统提示词:     500-800   tokens                        │
│ 知识块(20条):   400-1000  tokens                        │
│ 对话历史(20轮): 1600-2500 tokens                        │
│ 当前输入:       10-200    tokens                        │
│ ───────────────────────────────────────────────────────  │
│ 📊 预估成本:    2510-4500 tokens / chat() call          │
│ 📊 **平均值:    ~3200 tokens**                          │
└─────────────────────────────────────────────────────────┘
```

### 成本影响因素

| 因素 | 影响 | 可优化空间 |
|------|------|----------|
| 系统提示词大小 | 固定 (~700 tokens) | 改进和简化措辞 |
| 知识数量 | **线性增长** (N × 30-50) | ⭐⭐⭐ 最大优化点 |
| 对话历史长度 | **线性增长** (20轮 × 40) | ⭐⭐ 中等优化点 |
| 当前输入长度 | 较小 (~50-100) | 低优化影响 |

## 5️⃣ 现存问题和缺陷

### 问题 1: 无条件注入所有知识 ❌

**代码位置**: [server/ai/child.py](server/ai/child.py#L184-L188)

```python
relevant = await search_knowledge(session, user_text[:64])
seen_ids = {k.id for k in relevant}
for k in all_knowledge:
    if k.id not in seen_ids:
        relevant.append(k)
```

**问题**:
- 最终结果是 `relevant = all_knowledge` (搜索结果被覆盖)
- 即使有1000条知识，也会全部注入系统提示词
- 导致每次都浪费大量 token 在不相关的知识上

**影响**:
- 当知识库增长时，成本呈线性增长
- 系统提示词变得越来越长，可能超过模型的高效处理范围

---

### 问题 2: 搜索逻辑效率低 ❌

**代码位置**: [server/ai/memory.py](server/ai/memory.py#L80-L87)

```python
async def search_knowledge(
    session: AsyncSession, query: str
) -> List[KnowledgeItem]:
    result = await session.execute(
        select(KnowledgeItem).where(
            KnowledgeItem.topic.ilike(f"%{query}%")
            | KnowledgeItem.content.ilike(f"%{query}%")
        )
    )
    return result.scalars().all()
```

**问题**:
- 只用 `user_text[:64]` (前64字符) 做搜索 ← 太短，可能丢失关键词
- 使用模糊匹配 (`ILIKE`) 而非优化的全文搜索
- 没有按相关性排序

**改进方向**:
- 用完整的用户输入做关键词提取
- 实现向量相似度搜索 (embedding-based retrieval)
- 按置信度/重要性排序结果

---

### 问题 3: 没有知识过期或健忘机制 ❌

**观察**:
- 知识一旦存储就永久保留 (no TTL, no forgetting)
- 即使是低置信度的知识也列表形式而无优先级

**建议**:
- 为知识添加 `confidence` 字段（已在模型中有！但未使用）
- 只注入高置信度知识 (`confidence >= 70`)
- 实现遗忘机制：低置信度知识在一段时间后自动清理

---

### 问题 4: 没有缓存机制 ❌

**代码**: 每次调用 `chat()` 都执行完整的:
```python
all_knowledge = await get_all_knowledge(session)  # 每次都查 DB
relevant = await search_knowledge(session, user_text[:64])  # 每次都查 DB
```

**优化机会**:
- 缓存 `all_knowledge` (~频率: 仅当添加新知识时更新)
- 实现 LRU 缓存作为最近查询

---

### 问题 5: 没有优先级或重要性筛选 ❌

**现状**:
- 所有知识以 `[topic] content` 格式平等列出
- 没有利用 `KnowledgeItem.confidence` 字段
- 没有一个"最有用的知识"的概念

**改进**:
```python
# 应该：只注入高优先级知识
high_confidence = [k for k in all_knowledge if k.confidence >= 70]
sorted_by_relevance = sorted(high_confidence, 
                             key=lambda k: calculate_relevance(k.topic, user_text))
top_k_items = sorted_by_relevance[:10]  # 只注入Top 10最相关的
```

## 🔟 优化建议 (优先级排序)

### ⭐⭐⭐ 最高优先级 - 立即实施

1. **移除冗余的知识注入**
   - 问题: 搜索结果被覆盖
   - 方案: 修改循环逻辑，只注入 Top-K 最相关知识
   - 改动: ~10 行代码
   - 节省: **50-80% 的知识 token**
   
   ```python
   # 当前: relevant = all_knowledge (无效搜索)
   
   # 改为: 只注入 Top-5 最相关 + 按置信度排序
   relevant = sorted(relevant, 
                     key=lambda k: (k.confidence, -k.id))[:5]
   # 再加一些高置信度的背景知识
   background = [k for k in all_knowledge 
                if k.confidence == 100 and k.id not in {k.id for k in relevant}]
   relevant.extend(background[:5])
   ```

2. **按置信度和相关性筛选**
   - 问题: 所有知识被平等对待
   - 方案: 只注入 `confidence >= 70` 的知识
   - 改动: ~5 行代码
   - 效果: 过滤低质知识

3. **集成 PersonalityMemory 系统**
   - 问题: 性格记忆未被使用
   - 方案: 在 `_build_context()` 中调用 `build_personality_context()`
   - 改动: ~10 行代码
   - 效果: 加强身份一致性

---

### ⭐⭐ 中优先级 - 中期改进

4. **实现知识缓存**
   - 问题: 每次都查完整知识库
   - 方案: LRU 缓存 + 事件触发更新
   - 改动: ~50 行代码 + 缓存装饰器
   - 节省: **10-15% 的数据库查询时间**

5. **改进搜索逻辑**
   - 问题: 用前64字符搜索，精准度低
   - 方案: 关键词提取 + 向量相似度搜索
   - 改动: ~100 行代码 + 需要 embedding 模型
   - 效果: 提升搜索精准度到 80%+

6. **实现知识遗忘机制**
   - 问题: 知识永久保留
   - 方案: 标记过期知识，自动清理
   - 改动: ~50 行代码 + 后台任务
   - 效果: 保持知识库规模可控
